This document is for the system engineer who will add Attendance System permissions into the hosted central Auth system. The seed script is idempotent. It can be reviewed and run against CECB_Auth; it creates or updates the system, tasks, actions, task-actions, roles and role-permission mappings.

Setup Summary

  • Create or update central auth system: ATTENDANCE.
  • Create Attendance tasks and actions listed below.
  • Create task-action permissions listed below.
  • Create roles: ATTENDANCE_ADMIN, ATTENDANCE_AGM, ATTENDANCE_CLERK, ATTENDANCE_EMPLOYEE.
  • Assign users to the correct Attendance role.
  • For AGM and Clerk users, add UserWorkUnit assignments so the Attendance API can filter their assigned section/work unit data.
  • Configure Attendance API CentralJwtSettings.SecretKey with the same JWT secret used by central Auth.

Roles And Permissions

RoleCode Use For Permissions
ATTENDANCE_ADMIN Attendance admins. Full attendance data access and admin tools. All task-actions listed in this document. Expected active permission count: 26.
ATTENDANCE_AGM AGM users. They should see assigned work unit or section data only. DASHBOARD.VIEW_ASSIGNED, SECTION_ATTENDANCE.VIEW_ASSIGNED, EMPLOYEES.VIEW_ASSIGNED, REPORTS.VIEW_ASSIGNED, REPORTS.EXPORT, REPORTS.PRINT, OT_SUMMARY.VIEW_ASSIGNED, OT_SUMMARY.EXPORT, OT_SUMMARY.PRINT, ATTENDANCE_REGISTER.VIEW_ASSIGNED, ATTENDANCE_REGISTER.EXPORT, ATTENDANCE_REGISTER.PRINT, ANALYTICS.VIEW_ASSIGNED. Expected active permission count: 13.
ATTENDANCE_CLERK Clerk users. They should see assigned section/work unit data only. Same assigned-scope permissions as ATTENDANCE_AGM. Expected active permission count: 13.
ATTENDANCE_EMPLOYEE Normal employees. They should see their own attendance only. MY_ATTENDANCE.VIEW_OWN, REPORTS.VIEW_OWN, OT_SUMMARY.VIEW_OWN. Expected active permission count: 3.

Scope Rules

User Type Expected Data Scope Required Auth Setup
Admin All attendance data, all reports, user/settings menu. Assign ATTENDANCE_ADMIN.
AGM Only assigned work unit/section data. Assign ATTENDANCE_AGM and add UserWorkUnit rows.
Clerk Only assigned section/work unit data. Assign ATTENDANCE_CLERK and add UserWorkUnit rows.
Employee Only own attendance, own reports and own OT. Assign ATTENDANCE_EMPLOYEE and ensure EPF claim/mapping is available.
AGM and Clerk filtering depends on UserWorkUnit. Work unit IDs should match the ERP work space IDs used by Attendance employee data, such as AGMWorkSpaceId, DGMWorkSpaceId or ServiceUnitId. If a user has the role but no work unit assignment, they can log in but assigned data may be empty or limited.

Engineer Run Checklist

  • Backup or confirm safe deployment window for hosted CECB_Auth.
  • Open and review SeedAttendanceAuthPermissions.sql.
  • Update local default URLs in the script if hosted BaseUrl and UiUrl should be stored in Auth.
  • Run the script against hosted CECB_Auth.
  • Assign Attendance roles to test users.
  • Add UserWorkUnit rows for AGM and Clerk test users.
  • Set Attendance API CentralJwtSettings.SecretKey to the same JWT secret used by central Auth.
  • Restart Attendance API after configuration changes.
  • Test login as Admin, AGM, Clerk and Employee.

Verification Queries

Run these after the seed script to verify created permissions and role mappings.

SELECT s.SystemCode, s.SystemName, s.IsActive FROM dbo.[System] s WHERE s.SystemCode = N'ATTENDANCE'; SELECT t.TaskCode, a.ActionCode, CONCAT(N'ATTENDANCE.', t.TaskCode, N'.', a.ActionCode) AS PermissionCode FROM dbo.[System] s JOIN dbo.[Task] t ON t.SystemId = s.SystemId AND t.IsActive = 1 JOIN dbo.[TaskAction] ta ON ta.TaskId = t.TaskId AND ta.IsActive = 1 JOIN dbo.[Action] a ON a.ActionId = ta.ActionId AND a.IsActive = 1 WHERE s.SystemCode = N'ATTENDANCE' ORDER BY t.TaskCode, a.ActionCode; SELECT r.RoleCode, COUNT(rta.RoleTaskActionId) AS ActivePermissionCount FROM dbo.[System] s JOIN dbo.[Role] r ON r.SystemId = s.SystemId AND r.IsActive = 1 LEFT JOIN dbo.[RoleTaskAction] rta ON rta.RoleId = r.RoleId AND rta.IsActive = 1 WHERE s.SystemCode = N'ATTENDANCE' GROUP BY r.RoleCode ORDER BY r.RoleCode;

Application Behavior After Setup

  • Attendance UI reads roles and permissions from central login response/JWT and stores them as roleList and permissionList.
  • Sidebar and routes hide or show menus by permission.
  • Attendance API filters report and employee data by permission and user scope.
  • Admin receives all data. AGM and Clerk receive assigned data. Employee receives own data.
  • Existing attendance calculation rules, OT rules and check-in/check-out logic are not changed by this permission setup.
